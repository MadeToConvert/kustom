Overview

FixMyPhone accepts devices for repair without knowing the final cost upfront. The customer drops off their device, a technician assesses it, and the customer is charged only after approving the quoted price. This guide explains how to implement that flow using Kustom's recurring checkout feature on WooCommerce.

The flow has three stages: save the customer's payment method via a 0-amount checkout, assess and quote the repair, then charge the saved method once the customer confirms.

Customer books a repair
A 0-amount Kustom Checkout order is created with recurring: true. The customer enters their payment details and completes checkout — no charge is made.
Read token via Checkout API, assess device
When the push fires, read the completed order via the Checkout API to retrieve the recurring_token. Store it against the repair job. A technician inspects the device and generates a quote.
Customer approves — create a new order with the token
The customer approves the quote. FixMyPhone calls the Customer Token API with the stored recurring_token to create a new order. Capture is then handled via the Order Management API.
Order completed
The Customer Token API returns an order that can be acknowledged and managed via the Order Management API as normal.

Prerequisites

Required before starting
Recurring payments must be enabled on the FixMyPhone merchant account. Contact your Kustom integration manager to activate the feature flag.

Before implementing this flow, make sure the following are in place.

Requirement Details
Recurring feature enabled Must be activated by Kustom on the merchant account. The recurring flag in checkout requests is silently ignored if not enabled.
WooCommerce + KCO plugin Kustom Checkout for WooCommerce by Krokedil. Version 3.x or later recommended.
Server-side token storage A secure way to store the recurring_token per repair job — for example as a custom WooCommerce order meta field.
API credentials Kustom API username and password (used as HTTP Basic Auth) for the Customer Token API calls in step 3.
1

Create the booking order (0-amount, recurring)

When a customer books a repair, create a Kustom Checkout order with "order_amount": 0 and "recurring": true. The customer sees the checkout iframe, enters their payment details, and completes a zero-charge transaction. Kustom tokenises their payment method and returns a recurring_token on completion.

Checkout API request

Send a POST to https://api.kustom.co/checkout/v3/orders with Basic Auth. For a WooCommerce repair booking product, the payload looks like this:

{
  "purchase_country":  "SE",
  "purchase_currency": "SEK",
  "locale":            "sv-SE",
  "order_amount":      0,
  "order_tax_amount":  0,
  "order_lines": [
    {
      "type":             "physical",
      "reference":        "repair-booking",
      "name":             "Repair booking — payment authorisation",
      "quantity":         1,
      "unit_price":       0,
      "tax_rate":         0,
      "total_amount":     0,
      "total_tax_amount": 0
    }
  ],
  "recurring": true,
  "merchant_urls": {
    "terms":        "https://fixmyphone.se/terms",
    "checkout":     "https://fixmyphone.se/checkout",
    "confirmation": "https://fixmyphone.se/confirmation?order_id={checkout.order.id}",
    "push":         "https://fixmyphone.se/kustom/push?order_id={checkout.order.id}"
  }
}
Note
Even with order_amount: 0, the customer must complete the checkout flow. Their payment method is validated and tokenised at this point, but no charge is made.

The API returns an order_id and an html_snippet containing the checkout iframe. Render the snippet on the booking confirmation page. See Recurring payments overview for the full field reference.

2

Read the recurring token via Checkout API

After the customer completes the 0-amount checkout, Kustom sends a push notification to the merchant_urls.push endpoint. At this point the order status transitions to checkout_complete. Read the order via the Checkout API to retrieve the recurring_token.

Read the completed order — Checkout API

GET https://api.kustom.co/checkout/v3/orders/{order_id}
Authorization: Basic {base64(username:password)}

The response includes a recurring_token field at the top level:

{
  "order_id":        "ABC123",
  "status":          "checkout_complete",
  "recurring_token": "rct_XXXXXXXXXXXXXXXX",
  ...
}
Important
Store the recurring_token securely against the specific repair job — not just the customer. A customer may have multiple open repairs, each needing its own token.

Acknowledge the order — Order Management API

Once you've read and stored the token, acknowledge the 0-amount order via the Order Management API so Kustom knows it has been processed on your end.

POST https://api.kustom.co/ordermanagement/v1/orders/{order_id}/acknowledge
Authorization: Basic {base64(username:password)}

A successful acknowledgement returns HTTP 204 No Content. See Order Management API for full details.

3

Create a new order with the token after quote approval

Once the repair is assessed, communicate the price to the customer (email, SMS, or via the FixMyPhone portal). When they confirm, use the stored recurring_token to create a new order via the Customer Token API — no new checkout session is needed. The resulting order is then captured via the Order Management API.

Customer Token API request

POST https://api.kustom.co/customer-token/v1/tokens/{recurring_token}/order
Authorization: Basic {base64(username:password)}
Content-Type:  application/json

{
  "purchase_country":  "SE",
  "purchase_currency": "SEK",
  "locale":            "sv-SE",
  "billing_address": {
    "given_name":   "Anna",
    "family_name":  "Lindqvist",
    "email":        "anna@example.com",
    "phone":        "+46701234567",
    "street_address": "Storgatan 1",
    "postal_code":  "111 22",
    "city":         "Stockholm",
    "country":      "SE"
  },
  "order_amount":     149900,
  "order_tax_amount":  29980,
  "order_lines": [
    {
      "type":             "physical",
      "reference":        "screen-replacement-iphone15",
      "name":             "iPhone 15 screen replacement",
      "quantity":         1,
      "unit_price":       149900,
      "tax_rate":         2500,
      "total_amount":     149900,
      "total_tax_amount":  29980
    }
  ],
  "auto_capture": true
}
Amounts are in minor units
All amounts are in the lowest currency denomination. 149900 = 1 499,00 SEK. The tax_rate is expressed in hundredths of a percent — 2500 = 25%.

Setting "auto_capture": true captures the payment immediately. For a repair shop, it often makes more sense to omit this flag and capture manually via the Order Management API once the device is ready for collection. See Customer Token API reference.

Response

A successful response returns HTTP 200 with the new order object, including an order_id. Use this order_id with the Order Management API to capture, refund, or handle customer service on the order.

WooCommerce implementation

The KCO plugin handles the checkout iframe and push endpoint natively. You'll need custom code to set recurring: true, handle the 0-amount cart, and store the token. The recommended approach is a small custom plugin or a code snippet in functions.php.

Force 0-amount and recurring on the booking product

Use the kco_wc_api_request_args filter to modify the checkout request before it's sent to Kustom. Scope this to your repair booking product category so it doesn't affect other checkout flows.

/**
 * Set recurring: true and 0-amount for repair booking checkouts.
 * Runs only when the cart contains a product in the 'repair-booking' category.
 */
add_filter( 'kco_wc_api_request_args', function( $args ) {
    if ( ! fixmyphone_is_repair_booking_cart() ) {
        return $args;
    }

    $args['recurring']        = true;
    $args['order_amount']     = 0;
    $args['order_tax_amount'] = 0;

    // Replace order lines with a single 0-amount booking line
    $args['order_lines'] = [
        [
            'type'             => 'physical',
            'reference'        => 'repair-booking',
            'name'             => 'Repair booking — payment authorisation',
            'quantity'         => 1,
            'unit_price'       => 0,
            'tax_rate'         => 0,
            'total_amount'     => 0,
            'total_tax_amount' => 0,
        ],
    ];

    return $args;
} );

function fixmyphone_is_repair_booking_cart() {
    foreach ( WC()->cart->get_cart() as $item ) {
        if ( has_term( 'repair-booking', 'product_cat', $item['product_id'] ) ) {
            return true;
        }
    }
    return false;
}

Capture the recurring token on push

The KCO plugin processes the Kustom push and calls the woocommerce_checkout_order_created action. Hook into it to fetch the completed Kustom order and save the recurring_token as order meta.

/**
 * After KCO processes the push, fetch the Kustom order and save the
 * recurring_token to WooCommerce order meta.
 */
add_action( 'kco_wc_order_complete', function( $order_id, $kustom_order ) {
    $recurring_token = $kustom_order['recurring_token'] ?? null;

    if ( $recurring_token ) {
        update_post_meta( $order_id, '_kustom_recurring_token', $recurring_token );
    }
}, 10, 2 );
Hook availability
Check the exact hook name against your installed KCO plugin version. Older versions may use kco_wc_order_completed or expose the Kustom order data differently. The key is to capture the recurring_token from the Kustom order object at the time of push processing.

Charge via Customer Token API (backend)

When the customer approves the quote, trigger a server-side call to the Customer Token API. This is typically called from a custom WooCommerce admin action, a REST endpoint, or a webhook from your quoting system. Retrieve the token from order meta and send the request as shown in step 3.

/**
 * Example: Charge a repair order after customer approval.
 * Call this from your quote approval flow, passing the WooCommerce order ID
 * and the approved repair amount in øre/öre (minor units).
 */
function fixmyphone_charge_repair( int $wc_order_id, int $amount_minor, string $repair_description ) {
    $token = get_post_meta( $wc_order_id, '_kustom_recurring_token', true );

    if ( ! $token ) {
        throw new Exception( 'No recurring token found for order ' . $wc_order_id );
    }

    $credentials = base64_encode( KUSTOM_API_USER . ':' . KUSTOM_API_PASSWORD );
    $tax_amount  = (int) round( $amount_minor * 0.2 ); // 20% VAT example

    $response = wp_remote_post(
        'https://api.kustom.co/customer-token/v1/tokens/' . $token . '/order',
        [
            'headers' => [
                'Authorization' => 'Basic ' . $credentials,
                'Content-Type'  => 'application/json',
            ],
            'body' => wp_json_encode( [
                'purchase_country'  => 'SE',
                'purchase_currency' => 'SEK',
                'locale'            => 'sv-SE',
                'billing_address'   => fixmyphone_get_billing_address( $wc_order_id ),
                'order_amount'      => $amount_minor,
                'order_tax_amount'  => $tax_amount,
                'order_lines'       => [
                    [
                        'type'             => 'physical',
                        'reference'        => 'repair-' . $wc_order_id,
                        'name'             => $repair_description,
                        'quantity'         => 1,
                        'unit_price'       => $amount_minor,
                        'tax_rate'         => 2000, // 20% = 2000
                        'total_amount'     => $amount_minor,
                        'total_tax_amount' => $tax_amount,
                    ],
                ],
                'auto_capture' => true,
            ] ),
        ]
    );

    return json_decode( wp_remote_retrieve_body( $response ), true );
}

Error handling

The Customer Token API returns structured error objects when a charge fails. The two most common cases to handle are a declined or expired payment method and an invalid token.

Error code Meaning Recommended action
PAYMENT_METHOD_FAILED The saved payment method was declined, expired, or had insufficient funds. Notify the customer and ask them to update their payment method. The simplest approach is to create a new 0-amount recurring checkout (step 1) and replace the stored token.
TOKEN_EXPIRED The recurring_token has expired. Tokens are valid for 12 months from creation. Same as above — initiate a new booking checkout to capture a fresh token.
BAD_REQUEST Invalid or missing fields in the request body. Check that order_amount, order_lines, and billing_address are all present and correctly formatted. Amounts must be integers in minor units.
HTTP 401 Authentication failure. Verify that the API credentials are correct and that the Basic Auth header is properly base64-encoded as username:password.
Customer notification on payment failure
When PAYMENT_METHOD_FAILED occurs, send the customer a link back to a new repair booking checkout so they can enter updated payment details. Do not retry the charge on the same token — it will fail again.
